# Harness Engineering

> 为 AI Coding 时代构建可读、可控、可验证的软件工程系统

Harness Engineering 关注的，不只是“如何让 AI 生成代码”，而是：当 AI Agent 真正进入软件交付流程之后，软件工程系统本身应该如何演化。

从代码补全，到 Bug 修复、测试执行，再到自动生成 Pull Request，AI 正在逐步进入开发主流程。但问题也随之暴露出来：今天的大多数代码仓库、工具链和交付流程，本质上仍然是为人类开发者设计的。它们默认人能够理解隐性知识、遵守约定，并在评审中发现问题。可当 Agent 成为参与者，这些默认前提就不再成立。

Harness Engineering 的出发点正是这里：AI Coding 的瓶颈并不只是模型能力，而是工程系统是否具备让 Agent 安全参与、持续收敛的条件。代码库不再只是代码存储空间，而应当成为一个可以被机器读取、被规则约束、被反馈驱动的工作环境。

在这个意义上，Harness Engineering 主要回答三个问题：

- 如何提升系统可读性，让架构、领域知识与协作流程能够被 Agent 理解
- 如何建立防御机制，把工程约束变成系统边界，而不是依赖经验记忆
- 如何构建反馈回路，让 Agent 在生成、验证、修复中持续收敛，而不是一次性碰运气 

从防御视角看，Harness Engineering 也是一种治理框架。AI 并没有发明全新的漏洞类型，但它会放大缺陷扩散机制：一种“看起来合理”的错误实现，可能在多个模块、多个文件甚至多个仓库中被快速复制。因此，安全扫描、架构约束、受控接口、策略检查，不再只是交付后的补救措施，而应成为 AI 参与开发时的内建边界。

而当 Harness Engineering 继续往前推进，一个更关键的问题就会出现：系统如何判断 Agent 已经完成任务？传统软件工程里，“完成”往往是团队经验支撑下的共识；但在 Agent Loop 中，代码“看起来完成”并不等于系统层面的真正完成。Fitness Function 的意义，就在于把这些隐性的完成条件转化为可执行、可审计、可阻断的工程信号。任务不是因为通过了某一个局部检查而结束，而是因为满足了定义好的完成条件，才允许退出循环。:contentReference[oaicite:11]{index=11}

这个开源项目尝试系统化整理这套方法：  
从 Human-first 到 Agent-aware，  
从 Prompt 驱动到约束驱动，  
从“能生成”走向“可验证地完成”。

## 你会在这里看到

- Agent-aware 仓库与工程环境的设计原则
- 将隐性知识转化为机器可读上下文的方法
- 面向 AI 生成代码的防御与治理实践
- 用 Fitness Function 定义完成条件的工程模式
- Spec、Rules、Loop、Harness 协同工作的落地案例

## 核心观点

我们不应把 AI 当作软件交付流程中的魔法能力。

我们应该把它放进一个可读、可控、可验证、可治理的工程系统里。

这个系统，就是 Harness。
